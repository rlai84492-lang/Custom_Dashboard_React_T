# CampSyte 2.0 — Titan Campaign Cloud

A pixel-close React + Vite + Tailwind-style rebuild of the CampSyte 2.0 dashboard: Login, Overview, Campaigns
(list, create, detail with 8 sub-tabs — Details, Base Count, Segments, Creative, Approvals, Whitelisting,
Deployment, Performance), Calendar, Journey Builder, Customers, Library (brand templates + AI-generated
creative assets), Campground (AI chat workspace), Analytics (Performance / Operational), Leads & Follow-ups,
and Reports — all wired together with a persistent sidebar/header shell, a slide-over Copilot panel, working
light/dark mode, tasteful Framer Motion animation, and Excel + PDF export on every table.

Everything is mock data (see `src/data/mockData.js`) — there is no backend. All state (created campaigns,
lead status changes, theme, logged-in demo account, active brand) lives in React context and `localStorage`.
The primary demo dataset is **Titan Watches** (16 campaigns), with smaller Taneira and Mia datasets so every
demo account has non-empty data.

## Quick start

```bash
npm install
npm run dev      # generates styles.css, then starts the Vite dev server at http://localhost:3000
```

That's it — one command, one terminal. Open `http://localhost:3000` and log in with any account from the
"Demo accounts" panel (password is shown on screen: `Demo@CampSyte26`, though one-click login on any account
works instantly too). Try **Brand — Titan Watches** first; it has the richest dataset.

```bash
npm run build     # regenerates styles.css, then builds a production bundle to dist/
npm run preview   # serves the dist/ build locally to sanity-check the production build
```

## Project structure

```
index.html              Vite entry HTML (root-level, per Vite convention) — loads /src/main.jsx as a module
public/
  styles.css             generated by generate-css.js — do not hand-edit; Vite serves public/ at the site root
vite.config.js           Vite + @vitejs/plugin-react config (dev server on port 3000)
generate-css.js          hand-written Tailwind-equivalent utility-class generator (see "Styling" below)
src/
  main.jsx               ReactDOM root
  App.jsx                 route table (see "Routing" below)
  context/
    ThemeContext.jsx       light/dark mode, persisted to localStorage
    RouterContext.jsx      tiny hand-rolled client-side router (see below)
    AuthContext.jsx         current demo account + active brand + brand-switching
    DataContext.jsx         mutable slice of mock data (campaigns you create, lead status edits)
    CopilotContext.jsx      slide-over Copilot panel state + message thread
  components/              shared UI: Sidebar, Header, Layout, DataTable, Pagination,
                            StatCard, StatusBadge, ChannelChips, BrandChip, CopilotPanel/Fab,
                            CreateCampaignModal, icons.js (re-exports react-icons/md)
  pages/                    one file per sidebar destination, plus CampaignDetail.jsx
  data/mockData.js          every static dataset the app reads from
  utils/exportTable.js      CSV ("Export Excel") + real PDF (pdf-lib) export helpers
```

`build.js` and `qa-harness.html` are internal-only files used to smoke-test this project in the offline
sandbox it was authored in (which has no npm registry access, so Vite itself couldn't be installed there —
see "A note on how this was built" below). They are not part of the app and are safe to delete; they're
excluded from anything you'd deploy.

## Routing

There's no `react-router-dom` dependency — `src/context/RouterContext.jsx` is a ~90-line router built on
`history.pushState` / `popstate`, with a `matchPath('/campaigns/:id', path)` helper for the one dynamic
route. `App.jsx` is a plain if/else chain over `matchPath` calls, covering every sidebar destination plus
`/campaigns/:id`. It's intentionally small; swap in `react-router-dom` if the project grows and you'd
rather have nested routes / loaders — every `<Link>`/`navigate()` call already matches its API shape closely.

## Styling

`generate-css.js` is a Node script that writes `public/styles.css`: a hand-built stylesheet that mirrors
Tailwind's utility-class *naming conventions* (`flex`, `gap-4`, `rounded-2xl`, `bg-orange-600`,
`dark:bg-slate-900`, `hover:bg-orange-50`, `lg:grid-cols-3`, etc.) plus a small set of component classes
(`.btn`, `.card`, `.input`, `.badge`, `.tab-pill`, `.checklist-row`, `.asset-card`, `.phone-mock`, …) for the
recurring composite styles. It was written this way because the sandbox this project was originally built in
had no registry access to install the real `tailwindcss` package — functionally it behaves the same for
every class name actually used in `src/`.

**If you want the real Tailwind CLI/JIT compiler instead** (recommended once you have normal npm access —
which you do, since you're running this via Vite), it's a standard swap:

```bash
npm install -D tailwindcss postcss autoprefixer
npx tailwindcss init -p
```

Point its `content` config at `src/**/*.{js,jsx}`, replace the `<link rel="stylesheet" href="/styles.css">`
in `index.html` with your compiled Tailwind output (or import it from `main.jsx`), and delete
`generate-css.js` plus the `predev`/`prebuild` npm scripts. Class names in the JSX won't need to change since
they already follow Tailwind conventions — including the couple of arbitrary-value classes like
`max-w-[92%]` and `lg:grid-cols-[17rem_1fr]`, which real Tailwind supports natively.

If you add new class names to a component and see them unstyled, either add the utility rule to
`generate-css.js` and re-run `npm run css`, or switch to real Tailwind as above.

## Animation

Tasteful [Framer Motion](https://www.framer.com/motion/) animation is used throughout — staggered list/card
reveals (Overview priority list, Calendar bars, Library generated-asset grid, Segments cards), live-feed
style pulses (Analytics "Activity feed · LIVE", the green "live" dot, the notification badge), and
slide/fade transitions for dropdowns, the Copilot panel, and chat bubbles in Campground. It's applied with a
light touch — short durations (150–350ms), no bouncy overshoot, nothing on a loop that's distracting.

## Exports

- **Excel**: every "Excel" button downloads a UTF-8 CSV with a BOM, which Excel/Sheets/Numbers all open
  natively with no plugin. See `exportToExcel()` in `src/utils/exportTable.js`.
- **PDF**: every "PDF" button generates a real multi-page PDF client-side using `pdf-lib` (landscape A4,
  proportional column widths, automatic pagination, alternating row shading). See `exportToPdf()` in the
  same file.

Both work on every table in the app (Campaigns, Customers, Leads, Reports, Analytics audit trail), and every
table also has client-side pagination (`src/components/Pagination.jsx`) and search.

## Icons

`src/components/icons.js` re-exports a curated set of `react-icons/md` (Material Design icons — same icon
family Material UI ships) under semantic names (`IconOverview`, `IconCampaigns`, …) so pages never import
from `react-icons` directly. Every icon in the app comes from this file.

## Mock data & "backend"

`src/context/DataContext.jsx` holds the only mutable state: campaigns (creating one via the modal prepends
it to the list, tags it with your active brand, and auto-generates per-channel campaign IDs like
`TW-WA-710234`) and lead statuses (the dropdown in Leads updates in place). Everything else in
`src/data/mockData.js` — customers, segments, journeys, library templates + generated creative assets,
analytics numbers, approval audit trail, occasions, predictive insights, fatigue meter, base-count
spreadsheet rows, deployment checklist — is static and read-only. Wire up a real API by replacing the
imports in `DataContext.jsx` and the page components with fetch calls.

## A note on how this was built

This project was authored in a sandboxed environment with no npm registry access, so the original build used
a hand-rolled esbuild + custom-CSS toolchain instead of Vite/Tailwind. It has since been migrated to a real
Vite setup (this README, `package.json`, `vite.config.js`, and `index.html` all reflect that) — on a machine
with normal npm access (yours), `npm install && npm run dev` gives you the genuine Vite dev server with fast
refresh, and `framer-motion` installs for real. `build.js` / `qa-harness.html` are leftover from that
sandbox's internal smoke-testing and can be safely deleted.
